# Semantic Cache

## Why Not a Key-Value Cache

A traditional Redis string cache keys on the exact query text. This means:

- "What are the main risks in this document?" → cache miss
- "What risks does this document describe?" → cache miss (different string, same question)
- "List the key risks mentioned in this document" → cache miss (same again)

All three questions have identical intent and would produce identical answers. A key-value cache treats them as three separate uncached queries, making three LLM calls.

A semantic cache embeds the query into a 384-dim vector and searches for cached queries within cosine distance 0.08 (92% similarity). The second and third questions above would hit the cache — zero LLM calls.

## Implementation

`SemanticCacheEntry` model (in `query/models.py`):

| Field | Type | Purpose |
|-------|------|---------|
| `id` | UUID PK | — |
| `document` | FK → Document (CASCADE) | Cache entries are scoped to a document |
| `query_text` | TextField | Stored for debugging/admin display only |
| `embedding` | VectorField(384) | The embedded query — used for similarity search |
| `answer_json` | JSONField | `{"answer": "...", "citations": [...]}` |
| `created_at` | DateTimeField | Used for TTL filtering at read time |

The semantic cache lives in the same PostgreSQL instance as the rest of the application data — no separate cache service required. The `document` FK with `CASCADE` delete means removing a document automatically purges all its cache entries.

## The Similarity Threshold

This is the most technically precise part of the cache implementation.

pgvector's `<=>` operator returns **cosine distance**, not cosine similarity:
- Distance 0.0 = identical vectors
- Distance 1.0 = orthogonal vectors (unrelated)
- Distance 2.0 = opposite vectors

Cosine similarity is the complement: `similarity = 1.0 - distance`

We want a similarity threshold of 0.92 (92% similar). Converting:
```
distance_threshold = 1.0 - 0.92 = 0.08
```

The SQL generated by `SemanticCache.lookup()`:
```sql
SELECT id, answer_json, (embedding <=> %s) AS distance
FROM query_semanticcacheentry
WHERE document_id = %s
  AND created_at >= %s                      -- TTL filter
  AND (embedding <=> %s) <= 0.08            -- similarity threshold
ORDER BY distance ASC
LIMIT 1;
```

**Why 0.92:** This threshold was chosen empirically to catch paraphrases ("What are the risks?" ≈ "List the key risks") without false positives on topically related but distinct questions ("What are the risks?" vs "What are the opportunities?"). Tuning:
- Raise (e.g., 0.95 → distance 0.05): fewer cache hits, higher answer quality confidence
- Lower (e.g., 0.85 → distance 0.15): more cache hits, risk of returning a slightly wrong answer

## HNSW Index

`SemanticCacheEntry.embedding` has an HNSW (Hierarchical Navigable Small World) index, added in `query/migrations/0002_add_hnsw_index.py`.

```python
# The migration must be atomic=False because CREATE INDEX CONCURRENTLY
# cannot run inside a transaction. A regular migration would lock the table.
class Migration(migrations.Migration):
    atomic = False

    def database_forwards(self, app_label, schema_editor, from_state, to_state):
        schema_editor.execute(
            "CREATE INDEX CONCURRENTLY ..."
        )
```

HNSW provides approximate nearest-neighbor search in O(log n) rather than O(n) exact scan. The tradeoff: HNSW may miss a result that's technically within the distance threshold (it's approximate, not exact). The penalty for a false miss is one LLM call — the user gets a fresh answer, which is correct. This is acceptable.

## 7-Day TTL

Cache entries are not deleted eagerly. The TTL is enforced at **read time**:

```python
cutoff = timezone.now() - timedelta(days=SEMANTIC_CACHE_TTL_DAYS)  # 7 days
entry = SemanticCacheEntry.objects.filter(
    document_id=document_id,
    created_at__gte=cutoff,    # ignore entries older than 7 days
    ...
)
```

Old entries accumulate in the table and are simply filtered out. A background cleanup job (e.g., a periodic Celery beat task that deletes `created_at < cutoff`) is a future improvement. For now, the table grows but reads remain fast because the HNSW index is over the embedding column, and the `created_at` filter runs first.

`SEMANTIC_CACHE_TTL_DAYS` is set to 7 in `query/constants.py`. Rationale: documents don't change after ingestion, so stale answers within 7 days are acceptable. If a user re-uploads a changed document, the old document is deleted, which cascades and clears all its cache entries.

## Cache Invalidation

Two mechanisms, both automatic:

1. **Document delete:** `SemanticCacheEntry.document` is a FK with `on_delete=CASCADE`. Deleting a `Document` immediately deletes all its cache entries. No manual step needed.

2. **TTL expiry:** Entries older than `SEMANTIC_CACHE_TTL_DAYS` (7 days) are ignored on lookup. They remain in the DB but are never returned.

There is no manual cache invalidation endpoint. This is intentional — the system is designed so that the only valid reason to invalidate would be document content changing, which is handled by document deletion + re-upload + CASCADE.

## Failure Mode

Both `SemanticCache.lookup()` and `SemanticCache.store()` are wrapped in `try/except`:

```python
def lookup(self, query, document_id) -> dict | None:
    try:
        # ... pgvector query ...
    except Exception as e:
        logger.warning("Semantic cache lookup failed", extra={"error_type": type(e).__name__}, exc_info=True)
        return None  # treat as cache miss — LLM pipeline handles it

def store(self, query, document_id, answer_json) -> None:
    try:
        # ... create entry ...
    except Exception as e:
        logger.warning("Semantic cache store failed", extra={"error_type": type(e).__name__}, exc_info=True)
        # silently swallowed — user already received their answer
```

Any failure: lookup returns `None` (the query falls through to full retrieval + LLM), store logs a warning and returns. The user experience is unaffected — they get a correct answer either way. The cache is an optimization, not a dependency.

## Inspecting Cache Entries

**Via Django admin:** Navigate to `/admin/query/semanticcacheentry/`. Filter by document to see what questions are cached for it. The `query_text` field shows the original question.

**Via Django shell:**
```python
from query.models import SemanticCacheEntry
# See all cached entries for a document
entries = SemanticCacheEntry.objects.filter(document_id="your-doc-uuid").order_by("-created_at")
for e in entries:
    print(e.query_text, e.created_at)

# Manually clear cache for a document (useful in development)
SemanticCacheEntry.objects.filter(document_id="your-doc-uuid").delete()
```

**Checking why a cache miss occurred:** If you expected a cache hit but got a fresh LLM call, the likely causes are:
1. Distance threshold: the two queries are more than 0.08 apart (not similar enough)
2. TTL: the cached entry is older than 7 days
3. Document scope: the new query used a different `document_id`
4. Cache failure: an exception in `lookup()` (check logs for `"Semantic cache lookup failed"`)
